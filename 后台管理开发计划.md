# 后台管理端开发计划

## 1. 建设目标

当前网站已经完成前端页面搭建，后续要改为“前端只负责展示，数据统一由后台、API 和数据库提供”的结构。

核心目标：

- 首页、产品、质量保障、关于我们、联系我们/留言等页面内容不再写死在 HTML 中。
- 产品资料、产品分类、检测报告、关于我们文案、首页图片、联系方式、留言数据全部由数据库记录和管理。
- 前端页面不负责业务逻辑和计算，分类筛选、排序、显示/隐藏、图片绑定、“已投保”等展示规则，以及计算所需的参数和开关，都由数据库字段记录并由后端接口返回。
- 前端通过 API 获取数据并渲染页面。
- 后台管理端负责新增、修改、删除、排序、上下架和查看留言。
- 项目规模保持轻量，不引入复杂 CMS，不做商城、订单、支付。
- 目录结构清晰，前端、后台、API、数据库、上传文件分开管理。

## 2. 技术方向建议

本项目体量较小，建议使用：

- 后端：Node.js
- 数据库：SQLite
- API：REST API
- 后台页面：独立 admin 管理端
- 图片存储：本地 uploads 目录，数据库只保存图片信息和访问路径

SQLite 足够支撑当前企业展示站，不需要一开始就使用复杂数据库。后续如果数据量变大，可以再迁移到 MySQL。

## 3. 项目目录结构规划

建议后续整理为以下结构：

```text
紫砂网站/
  frontend/
    pages/
      index.html
      products.html
      quality.html
      about.html
      contact.html
    assets/
      css/
      js/

  admin/
    pages/
    assets/
      css/
      js/

  api/
    routes/
    controllers/
    validators/

  backend/
    server.js
    services/
    repositories/
    middleware/
    utils/

  database/
    schema.sql
    migrations/
    seeds/
    site.sqlite

  uploads/
    images/
    reports/

  后台管理开发计划.md
```

说明：

- `frontend/`：只放用户看到的页面和前端渲染脚本。
- `admin/`：只放后台管理页面，不和前台页面混在一起。
- `api/`：只放接口路由、参数校验和控制器。
- `backend/`：只放业务逻辑、数据库访问、登录权限等服务代码。
- `database/`：只放数据库结构、初始化数据和数据库文件。
- `uploads/`：只放后台上传的图片、报告图和其他媒体文件。
- `后台管理开发计划.md`：按当前要求保留在项目根目录。

当前已有的 HTML 页面可以先保留，后续迁移时逐步移动到 `frontend/pages/`。其中现有 `message.html` 后续建议统一命名为 `contact.html`，页面含义更贴合“联系我们/留言”。

## 6. 后台管理端模块

后台建议一次性完成以下模块，不做复杂权限系统，不按角色分配菜单和接口权限。

1. 登录页
   - 管理员账号密码登录。
   - 登录后才能访问后台。

2. 首页管理
   - 管理首页轮播图。
   - 管理首页文案、模块显示状态和排序。

3. 产品分类管理
   - 新增、修改、删除分类。
   - 调整分类横向展示顺序。

4. 产品管理
   - 管理产品名称、图片、简介、分类、排序。
   - 控制产品是否显示。
   - 控制是否显示“已投保”标识。

5. 质量保障/检测报告管理
   - 管理七类检测报告标题。
   - 每个报告下绑定多张报告图片。
   - 支持图片排序，保证报告和图片一一对应。

6. 关于我们管理
   - 管理公司介绍、发展信息、企业文化、配图。
   - 通过 `pages` 和 `page_sections` 维护，不再单独建立重复的关于我们表。
   - 所有未核实资料必须标记为待确认，不在前台展示虚假信息。

7. 联系方式管理
   - 管理电话、邮箱、地址、微信等基础信息。
   - 直接维护 `site_settings` 表，不单独新增复杂联系信息表。

8. 留言管理
   - 查看客户留言。
   - 修改处理状态。
   - 添加后台备注。

9. 媒体库
   - 上传产品图片、报告图片、关于我们图片。
   - 统一查看和复用图片。

## 7. 前端改造原则

前端页面后续只保留结构和样式，不直接写死业务数据。

改造原则：

- `index.html` 不写死轮播、产品和公司信息，通过 API 获取。
- `products.html` 不写死产品分类和产品卡片，通过 API 获取。
- `quality.html` 不写死检测报告文字和图片，通过 API 获取。
- `about.html` 不写死公司介绍内容，通过 API 获取。
- `contact.html` 表单提交到 `POST /api/messages`，留言进入数据库。
- header 导航、footer 联系信息从统一接口读取，避免每个页面重复维护。

## 8. 图片与文件管理原则

- 所有后台上传文件放入 `uploads/`。
- 产品图、报告图、关于我们配图等业务图片不放在前端目录中。
- 产品图片建议放入 `uploads/images/products/`。
- 检测报告图片建议放入 `uploads/reports/`。
- 关于我们图片建议放入 `uploads/images/about/`。
- 数据库保存文件路径和图片说明，不把图片本身存进数据库。
- 删除图片前要检查是否被产品、报告或页面使用，避免误删正在展示的图片。

图片读取逻辑：

- 产品表、报告图片表、页面模块表只保存 `image_id`。
- `media_files.file_path` 保存真实访问路径，例如 `/uploads/images/products/example.jpg`。
- API 查询数据时把业务表和 `media_files` 关联，返回前端可直接使用的 `image_url`。
- 前端只使用 API 返回的 `image_url` 渲染图片，不在 HTML 中写死图片地址。

## 9. 命名规范

### 9.1 目录命名

- 使用英文小写。
- 多个单词使用中横线或下划线，但同一项目内保持统一。
- 示例：`product-categories`、`quality-reports`。

### 9.2 数据库字段命名

- 使用英文小写加下划线。
- 字段名必须表达真实含义。
- 示例：`product_interest`、`sort_order`、`is_visible`、`created_at`。

### 9.3 API 命名

- 使用 REST 风格。
- 列表资源使用复数名词。
- 示例：`/api/products`、`/api/product-categories`、`/api/quality-report-groups`。

### 9.4 前端变量命名

- JavaScript 使用小驼峰。
- 示例：`productList`、`reportGroups`、`siteSettings`。

## 10. 一次性开发完成清单

本项目不按阶段拆分交付，开发时一次性完成前端数据化、API、数据库和 admin 后台。以下是必须同时完成的工作清单。

### 10.1 数据库和基础后端

- 建立数据库结构。
- 建立初始化数据。
- 建立后端服务入口。
- 建立数据库访问层。
- 建立上传文件目录和媒体文件记录逻辑。

### 10.2 公开 API

- 完成首页、产品、质量报告、关于我们、联系我们/留言等公开接口。
- 用当前静态页面已有内容作为第一批种子数据。
- 所有前端页面通过 API 获取数据。

### 10.3 前端数据化

- 把静态页面中的硬编码内容替换为 API 渲染。
- 首页、产品页、质量报告页、关于我们页、留言页全部接入接口。
- header 导航和 footer 联系方式接入统一接口。

### 10.4 后台管理端

- 完成管理员登录。
- 完成产品、分类、报告、图片、关于我们、基础设置、留言管理。
- 每个模块完成新增、修改、删除、排序、显示/隐藏。
- 后台只区分“未登录”和“已登录管理员”，不做角色权限配置。

### 10.5 数据校验和安全

- 留言表单做必填校验。
- 上传文件限制格式和大小。
- 后台接口必须登录后访问。
- 管理员密码必须加密存储。

### 10.6 测试和上线准备

- 检查所有前台页面是否能从 API 正常渲染。
- 检查后台修改后前台是否同步变化。
- 检查检测报告图片是否和报告标题一一对应。
- 检查留言是否成功进入数据库。
- 建立数据库备份方案。

## 11. 暂不开发内容

当前项目不做以下功能：

- 在线支付
- 购物车
- 订单系统
- 会员系统
- 多管理员复杂权限
- 后台角色权限分配
- 多语言站点
- 复杂 CMS 模板编辑器

这些功能会增加复杂度，不符合当前企业展示站的实际需求。

## 12. 核心需求总结

本次后台化改造的核心不是重做页面，而是把现有静态页面的数据抽出来，统一交给数据库、API 和后台管理端维护。

最重要的要求：

- 前端页面负责展示，不再硬编码产品、报告、图片、文案和留言数据。
- 数据库负责存储产品、分类、报告、图片记录和路径、页面内容、联系方式和留言。
- API 负责把数据库数据传给前端。
- admin 后台负责让管理员维护所有页面数据。
- 项目目录必须清晰分层，不能把前端、后台、接口、数据库代码混在一起。
- 表名、字段名、接口名必须语义明确，后续维护时一眼能看懂。
